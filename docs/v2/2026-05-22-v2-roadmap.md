# Raven v2.0 — Compiled, OOP-via-Traits, GitHub-Native Packages

> **Status:** Plan for review. This is a roadmap, not a step-by-step implementation guide — v2.0 is a multi-quarter project that must be broken into smaller specs/plans per phase. Approving this plan greenlights the *direction* and the *branching/release strategy*. Each phase below will get its own spec + plan before code lands.

## Context

Raven's stated design pillars are: **performance and control of C++, safety and modern features of Rust, readability of Python, structure of Java, simplicity of Go.** v1.6.4 is a tree-walking interpreter that hits the Python/Java/Go pillars partially but misses the C++ and Rust ones entirely (no native compilation, no traits, no sum types, no generics). v2.0 is the redesign that closes the gap.

### How v2 stacks up against the pillars

| Pillar | v2 design status | Notes |
|---|---|---|
| Safety of Rust | ✅ Fully | Static types + tracing GC = memory safe. Different mechanism than Rust's borrow checker, same end result for user code. |
| Modern features of Rust | ✅ Fully | Traits, sum types with payloads, pattern matching, generics, `Result<T, E>` / `Option<T>`, type inference. |
| Readability of Python | ✅ Fully | Local type inference, string interpolation, no semicolons, range-based `for`, lambdas, single-expression function bodies. |
| Structure of Java | ✅ Fully | Static types, modules, structs-with-methods, traits for polymorphism, exhaustive `match`. (Note: no visibility modifiers — see "Visibility" section. Structure comes from the type system, not access control.) |
| Simplicity of Go | ✅ Fully | Small core, tracing GC, GitHub-direct packages, composition over inheritance, no visibility ceremony. |
| Performance of C++ | ✅ via Cranelift + tuned GC | Native machine code via Cranelift. GC and allocation overhead are real but the claim stands as long as we tune carefully (avoid unnecessary allocations in hot paths, generational GC as a v2.x optimization). For inner-loop work users can drop to C via the FFI. |
| Control of C++ | ✅ via C FFI | C FFI is **non-negotiable** in v2.0 scope — it is what makes this pillar truthful. Users who need raw memory layout, manual allocation, or system calls drop to C through `extern "C"` and call it from Raven. |

All seven pillars delivered. README copy ("performance and control of C++, safety of Rust, ...") stays as-is; v2.0's job is to make it true.

**Critical consequence:** because we are not softening the README, **C FFI is a hard scope requirement, not optional.** The v2.0 release cannot ship without it.

### Feature heritage — what to take, what to reject from each parent

The pillars name five languages. Surveying their reception, here's what Raven v2 keeps and what it deliberately rejects. The goal: combine the loved features, cut the painful ones.

#### C++

| Take | Reject |
|---|---|
| Native machine-code performance | Header files / textual `#include` |
| Zero-cost abstractions where possible | Undefined behavior (UB) as a language concept |
| Deterministic resource cleanup (we get this via `defer` — see below) | Manual memory management (use-after-free, leaks) |
| Multi-paradigm flexibility | Multiple inheritance & the diamond problem |
| Direct C interop (our FFI) | Most-vexing-parse and other parser ambiguities |
| | Slow compile times caused by template instantiation in headers |
| | Build-system culture (CMake, autoconf) |
| | ABI instability |

#### Rust

| Take | Reject |
|---|---|
| Sum types + exhaustive `match` | Borrow checker / lifetime annotations (we use GC) |
| `Result<T, E>` and `Option<T>` instead of exceptions / null | "Function color" pain from `async`/`.await` |
| `?` operator for error propagation | Slow compile times from monomorphization explosions (we'll watch this) |
| Traits as the polymorphism mechanism | Macro complexity (we're not shipping macros in v2.0) |
| Helpful error messages with source pointers (already from v1's `RavenError`) | Trait coherence / orphan rules ceremony |
| `Cargo`-style first-party package manager (our `rvpm`) | Turbofish (`::<T>`) — we'll allow `<T>` in expression position |
| `cargo fmt` / `rustfmt` canonical formatting | Trait-object verbosity (`dyn Trait` everywhere) — we default to static dispatch for generics |

#### Python

| Take | Reject |
|---|---|
| Readable, low-noise syntax | Dynamic typing (runtime type errors that should be compile-time) |
| String interpolation (`"hello, ${name}"` — our take on f-strings) | GIL / global interpreter lock |
| List/dict literals as primary syntax (`[1,2,3]`, `["a": 1]`) | Significant whitespace (we use braces) |
| Single-expression function bodies | Mutable default argument gotcha |
| List comprehensions in spirit — provided via method chaining (`xs.map(...).filter(...)`) rather than a new syntax | Multiple inheritance + MRO confusion |
| Plain-language standard library naming | Versioning hell (Py2 vs 3) — we commit to strict semver within v2.x |
| | `self` written explicitly on every method (we keep it but only as the first param, not in calls) |
| | `__dunder__` proliferation |

#### Java

| Take | Reject |
|---|---|
| Static typing for refactorability and tooling | Verbosity (`FooFactory factory = new FooFactory()`) |
| Strict backward-compatibility ethic within a major version | `null` everywhere — we have `Option<T>` instead |
| Predictable, conservative evolution | NullPointerException as a runtime fact of life |
| Strong IDE/tooling alignment (we plan good VS Code support) | Checked exceptions — we use `Result` |
| | Type erasure on generics — we monomorphize like Rust |
| | XML configuration culture |
| | Property boilerplate (getters/setters) — fields are direct |
| | JVM startup time |
| | Build-system fragmentation (Maven vs Gradle vs ...) |

#### Go

| Take | Reject |
|---|---|
| Fast compile times (Cranelift gives us this) | `if err != nil { return err }` boilerplate — we have `?` |
| Single static binary output | `interface{}` as a typing escape hatch — we have proper generics |
| Goroutine-simple concurrency model (deferred to v2.x; not v2.0) | No sum types — we have enum-with-payloads |
| GitHub-direct package paths | Nil-interface gotcha (`(T)(nil) != nil`) |
| `gofmt`-style one-true-formatter (our `rvpm fmt` is non-negotiable) | Mandatory unused-import errors as compile errors (we'll warn, not fail) |
| Module simplicity (no central registry, content-hashed lockfile) | Capitalization-based visibility — we use no visibility modifier instead |
| Tooling built into the language CLI | No exhaustive switch — our `match` is exhaustive |
| `defer` for cleanup (we adopt this — see below) | No default function argument values — see "open question" #4 |

### Net additions to the language driven by this analysis

The heritage review surfaces concrete features the original design needs. Adding them to scope:

1. **`defer` statement** (from Go, replaces C++ RAII). Schedules a callback to run at scope exit, in LIFO order. Used for file close, lock release, etc.
   ```rust
   fun read_config() -> Result<Config, IoError> {
       let f = open("config.toml")?
       defer f.close()
       parse(f.read_all())
   }
   ```
   This is how we deliver C++-style "deterministic cleanup" without RAII destructors or `Drop` traits.

2. **Static dispatch by default for generics; `dyn Trait` is an explicit opt-in.** Rust forces `dyn` annotations in many places where users find it noisy. Raven's rule: a generic parameter `<T: Trait>` always monomorphizes (static dispatch); `dyn Trait` is only used as a value/field type when you need a heterogeneous collection.

3. **No turbofish.** Type arguments at call sites use plain `<T>`: `parse<Int>("42")`, not `parse::<Int>("42")`. Grammar will be designed to make this unambiguous.

4. **Method chaining over comprehensions.** Skip Python-style `[x*2 for x in xs if x > 0]` syntax. Provide `.map`, `.filter`, `.collect` (or `.toList`) as iterator methods. Reads as well, one less grammar form.

5. **One canonical formatter** (gofmt principle). `rvpm fmt` is the only blessed style. No style options, no flags. CI enforces it.

6. **Unused-import / unused-variable: warnings, not errors.** Go-style hard failure on unused imports is widely disliked. We warn at compile time. CI lints can promote to errors per-project.

7. **No `null`, anywhere.** `Option<T>` and `T?` sugar are the only absent-value mechanism. The word "null" doesn't exist in the language.

8. **Strict semver discipline within v2.x.** Every v2.x release is backwards-compatible at the source level. Behavior changes that aren't strictly additive wait for v3.

9. **Compile times are a feature.** A non-trivial program (~10k lines, a dozen dependencies) compiles in well under 5 seconds debug, well under 30 seconds release. CI tracks this as a benchmark and regressions block merges.

10. **Error messages with hints, like Rust.** `RavenError` already has a `hint: Option<String>` field from v1; v2 keeps that and uses it liberally. Every common error should suggest the fix.

### Why v2 (concrete drivers)

The just-merged maintainability refactor (`code_gen/`, `type_checker/` directory modules) cleaned up the interpreter, but the underlying execution model — and the language itself — has limits that prevent Raven from being a serious general-purpose tool:

- **Performance:** tree-walking the AST per call is fundamentally slow. Compiled native code is 10-100× faster.
- **Language gaps the v1 design locked in:** no generics, no traits/interfaces, no sum types (enum payloads), no pattern matching, no string interpolation, C-style for-loops only, no compound assignment, sentinel-value error handling instead of proper `Result`.
- **OOP is anemic:** structs + impl blocks exist, but with no polymorphism, no protocol/interface mechanism, and no encapsulation. Collections in `lib/collections.rv` are hard-coded to `string[]` payloads because there are no generics.
- **Package manager (`rvpm`) is a stub:** `install`/`add` exit with "not yet implemented." Modules resolve from the filesystem only.

v2.0 is a deliberate, full breaking change to fix all of the above in one coherent redesign. The user chose:

- **Compilation target:** Cranelift backend (pure-Rust, no LLVM dependency).
- **Memory model:** tracing garbage collection (Go-like simplicity, no ownership/lifetime annotations).
- **OOP shape:** structs + traits/protocols, composition over inheritance (Rust/Go-style).
- **Package manager:** Go-style GitHub-direct imports — no central registry, just `import "github.com/user/repo/pkg"` with content-addressed local cache and an `rv.lock` file.

The intended outcome: a clean, modern, statically-typed compiled language that produces native binaries via Cranelift, with a familiar GC'd object model, ergonomic OOP via traits, and a frictionless package ecosystem that piggybacks on GitHub.

## Goals & non-goals

### Goals
1. Single static binary output per program (Cranelift → object → system linker).
2. GC'd runtime with no manual memory management in user code.
3. Static type system with generics, traits, sum types (enums with payloads), and pattern matching.
4. Local-type inference (`let x = 5` infers `Int`).
5. First-class functions and closures.
6. Ergonomic OOP via `struct` + `trait` + `impl Trait for Struct`.
7. String interpolation, range-based `for`, compound assignment, proper `Result<T,E>` and `Option<T>`.
8. `rvpm` fetches packages directly from GitHub using a Go-style import path.
9. **C FFI** — declare external C functions, link against system C libraries. This is what makes the "control of C++" pillar honest.
10. **`defer` statement** for deterministic resource cleanup (RAII without destructors).
11. **One canonical formatter** — `rvpm fmt` enforced, no style flags, gofmt-style.
12. **Fast compile times as a non-functional requirement** — ~10kloc program builds debug in < 5s, release in < 30s on a typical laptop. Tracked in CI.
13. **No `null`** — `Option<T>` only.
14. **Strict semver within v2.x** — every minor/patch release source-compatible with prior v2.x releases.
15. Self-hosted-friendly: the compiler stays in Rust for v2, but the language must be expressive enough to host its own future rewrite.
16. Migration path for users: v1.x stays maintained on a `v1.x-maintenance` branch for critical fixes; v2 is a deliberate breaking change with no automated migration tool — a written migration guide instead.

### Non-goals
- **Backwards compatibility with v1 syntax.** This is a clean break.
- **Macros / metaprogramming.** Out of scope for v2.0. Maybe v2.x.
- **Async / concurrency primitives.** Out of scope for v2.0. Goroutines-style scheduler is a v2.x discussion.
- **Self-hosting the compiler.** v2.0 ships with the compiler still in Rust.
- **A central package registry.** GitHub-only, like Go.
- **LLVM backend.** Cranelift only. (Could be added later if performance demands it.)
- **JIT.** AOT only.
- **Inheritance, multiple inheritance, mixins.** Composition + traits only.
- **REPL.** Compiled languages typically don't have one; can revisit later.

## v2 language design (overview)

A snapshot of the syntax. Every detail here is open to revision during phase-level specs, but this is the target shape.

### Variables, types, primitives

```rust
let x = 5                  // Int inferred; mutable
let pi: Float = 3.14       // explicit type when needed
let counter = 0            // mutable by default
counter += 1               // compound assignment

let name: String = "Raven"
let flag: Bool = true

const MAX_USERS: Int = 100 // compile-time constant; immutable
```

Variables declared with `let` are mutable. There is no `var` keyword and no `let mut` ceremony — every `let` binding can be reassigned. `const` is reserved for compile-time constants only (literal values evaluatable at compile time).

Primitives: `Int` (i64), `Float` (f64), `Bool`, `String`, `Char`, plus `()` (unit). No implicit conversions. Type names PascalCase for built-ins and user types alike (a deliberate break from v1's `int`/`string` lowercase).

### Strings with interpolation

```rust
let greeting = "Hello, ${name}!"           // ${expr} interpolation
let multiline = """
    block string,
    no escaping needed
"""
```

### Collections

```rust
let nums: List<Int> = [1, 2, 3]
let scores: Map<String, Int> = ["a": 1, "b": 2]
let unique: Set<String> = {"x", "y"}
```

Generic `List<T>`, `Map<K, V>`, `Set<T>` in stdlib. No raw arrays at the language level for user code.

### Functions, closures

```rust
fun add(a: Int, b: Int) -> Int = a + b     // single-expression body

fun greet(name: String) -> String {
    return "Hi, ${name}"
}

let double = fun(x: Int) -> Int = x * 2    // closure
let triple = { x -> x * 3 }                 // shorthand lambda
```

### Control flow

```rust
for x in 0..10 { print(x) }                 // range
for item in list { print(item) }            // iterator
for (i, item) in list.enumerate() { ... }   // destructured

while cond { ... }
loop { ...; break }                          // infinite loop with break

if x > 0 { ... } else if x == 0 { ... } else { ... }
// `if` is an expression: let s = if x > 0 { "pos" } else { "neg" }
```

No semicolons. Newlines and braces handle separation. C-style for-loops are gone.

### Deterministic cleanup with `defer`

```rust
fun process(path: String) -> Result<Int, IoError> {
    let f = File.open(path)?
    defer f.close()             // runs at scope exit, LIFO order
    let lock = mutex.acquire()
    defer lock.release()
    do_work(f)
}
```

`defer` schedules a callback that runs when the enclosing function (or block, TBD in phase 1 spec) returns — normally, via `?` propagation, or via `panic`. Multiple `defer` statements run in reverse declaration order. Replaces RAII destructors and avoids the need for `Drop` trait machinery.

### Structs, traits, impls

```rust
struct Point {
    x: Float
    y: Float
}

trait Display {
    fun show(self) -> String
}

trait Distance {
    fun distance(self, other: Self) -> Float
}

impl Display for Point {
    fun show(self) -> String = "(${self.x}, ${self.y})"
}

impl Distance for Point {
    fun distance(self, other: Point) -> Float =
        ((self.x - other.x).pow(2) + (self.y - other.y).pow(2)).sqrt()
}

// Use:
let p = Point { x: 1.0, y: 2.0 }
print(p.show())
```

Traits are the only polymorphism mechanism. `dyn Trait` is the trait-object syntax for dynamic dispatch when needed. No inheritance.

### Sum types + pattern matching

```rust
enum Shape {
    Circle(radius: Float)
    Rectangle(width: Float, height: Float)
    Triangle(a: Float, b: Float, c: Float)
}

fun area(s: Shape) -> Float =
    match s {
        Circle(r) -> 3.14159 * r * r
        Rectangle(w, h) -> w * h
        Triangle(a, b, c) -> {
            let s = (a + b + c) / 2.0
            (s * (s-a) * (s-b) * (s-c)).sqrt()
        }
    }
```

`match` is exhaustive at compile time. Variants can have named or positional payloads.

### Optionals and Results

```rust
let maybe: Int? = some_lookup(key)         // sugar for Option<Int>
match maybe {
    Some(n) -> print(n)
    None    -> print("missing")
}

fun parse_int(s: String) -> Result<Int, ParseError> { ... }
// Use:
let n = parse_int("42")?                   // ? propagates Err, unwraps Ok
```

No more sentinel `0` returns or empty-string failures. Failure is in the type signature.

### Modules and imports

```rust
import std/io                              // stdlib module
import std/collections/{Map, Set}          // selective
import github.com/martian56/raven-http     // third-party from GitHub
import http "github.com/martian56/raven-http" as h   // aliased
```

`std/...` paths are reserved for built-in stdlib. Anything with a `host.tld/...` prefix is fetched by `rvpm`. Local-relative paths (`./foo`) still work for in-project modules.

### C FFI

```rust
extern "C" {
    fun strlen(s: CStr) -> Int
    fun open(path: CStr, flags: Int) -> Int
}

let n = strlen(c"hello")   // c"..." is a null-terminated C string literal
```

`CStr`, `CInt`, `CPtr<T>` etc. are primitive types in `std/ffi`. The codegen emits direct C-ABI calls; linking is handled by `rvpm build`'s pass-through of `-l<lib>` flags from `rv.toml`.

### Visibility

**No visibility modifier.** Every top-level declaration (`fun`, `struct`, `trait`, `enum`, `let`, `var`) is importable from other modules. Struct fields are publicly readable and writable. This drops v1's `export` keyword entirely.

Rationale: keeps the language clean and matches the "simplicity of Go" pillar — module-level identifiers are part of the module's API, full stop. Users who want to signal "internal" can use a naming convention (e.g. leading underscore `_helper_fn`) but it's documentation, not enforcement.

```rust
fun greet(name: String) -> String { ... }       // automatically importable
struct User { name: String, age: Int }          // automatically importable
```

## Compiler architecture

Mirrors rustc's progressive lowering, scoped to Raven's much smaller surface:

```
Source (.rv)
   │
   ▼  Lexer            tokens with spans
   ▼  Parser           AST (concrete syntax tree)
   ▼  Resolver         binds names to declarations; resolves imports via rvpm
   ▼  Type Checker     inference + generics + trait resolution; produces typed-AST
   ▼  HIR lowering     desugars: for→loop+iter, ?→match, ${}→concat, etc.
   ▼  MIR lowering     basic blocks, explicit control flow, monomorphized generics
   ▼  Codegen          Cranelift IR per function → object file
   ▼  Linker           system `cc`/`link` → native binary
```

Per-module files in the new `src/v2/` tree (or whatever directory layout the phase-1 spec chooses — likely a fresh `src/` after the v2 cutover):

| Stage | Module |
|---|---|
| Lexer | `lexer/` (port of v1 with minor token additions: `var`, `trait`, `impl`, `match`, `pub`, range `..`, `..=`, `?`, etc.) |
| Parser | `parser/` (new grammar; recursive descent following v1's pattern) |
| AST | `ast/` (much richer than v1: generic params, trait bounds, match arms, etc.) |
| Resolver | `resolve/` (NEW: walks the AST, binds identifiers, resolves imports through `rvpm`) |
| Type checker | `tycheck/` (generics + trait inference + exhaustiveness check on `match`) |
| HIR | `hir/` (NEW) |
| MIR | `mir/` (NEW; basic blocks + monomorphization) |
| Codegen | `codegen/` (NEW; wraps `cranelift-codegen` + `cranelift-module`) |
| Driver | `driver/` (NEW; orchestrates the pipeline; replaces v1's `main.rs` orchestration) |
| Runtime | separate crate `raven-runtime/` linked into compiled binaries |

**v1 source is deleted from `v2.x` immediately at branch creation (Phase 0).** Reference to v1 semantics during v2 development is via the v1 docs (`docs/syntax.md`, `lib/*.rv`) and the `v1.x-maintenance` branch — not by keeping v1 source alongside v2. This keeps the v2.x tree clean and forces the v2 design to stand on its own. The v1 docs stay in `docs/` indefinitely as historical reference.

## Runtime architecture

A small Rust crate `raven-runtime/` linked statically into every compiled program. Provides:

1. **GC** — Initial implementation: a conservative or precise mark-and-sweep collector. Allocation goes through a runtime alloc function emitted into the binary. Roots discovered via stack-map metadata Cranelift emits (or a shadow-stack approach as a simpler v1 implementation).
2. **Built-in object layouts** — String, List, Map, Set, closures, trait objects. Each has a known header (type tag + GC bits + length/capacity as appropriate).
3. **Runtime panic/abort** — `panic("msg")` prints, dumps stack trace, exits non-zero.
4. **Stdlib hooks** — fs/net/http/time/io are runtime functions called from the generated code via plain C ABI symbols. Most are thin wrappers over `std::*` or `ureq`.
5. **Trait object dispatch** — fat pointers (data + vtable), like Rust's `dyn Trait`.

GC strategy: start with simplest viable (stop-the-world mark-and-sweep with shadow stack for roots). Generational/incremental is a v2.x optimization once the language is stable.

## rvpm v2 — Go-style packages

### Import path → fetch
- `import github.com/martian56/raven-http` resolves by:
  1. Looking up `~/.rvpm/cache/github.com/martian56/raven-http@<version>/` locally.
  2. If absent, cloning via `git clone --depth 1 --branch <tag> https://github.com/martian56/raven-http` into the cache.
  3. Version pinning via `rv.lock` (content hash of the resolved tree).
- Versions are git tags following semver (`v1.2.3`). If `rv.toml` has no version constraint, the latest tag wins on first resolve and gets pinned in `rv.lock`.
- No central registry. No publishing step — pushing a tag to GitHub *is* the publish.

### Files
- `rv.toml` — declares the package + its direct dependencies. New `[dependencies]` syntax:
  ```toml
  [dependencies]
  "github.com/martian56/raven-http" = "1.0"
  ```
- `rv.lock` — checked into the repo (like `Cargo.lock` for applications, `go.sum` for Go libraries too). Stores exact versions + tree hashes for reproducible builds.
- `~/.rvpm/cache/<host>/<user>/<repo>@<version>/` — read-only cached source. Shared across all projects.

### Subcommand surface
- `rvpm init [name]` — scaffold `rv.toml` + `src/main.rv`. (Already exists in v1; minor update for new manifest format.)
- `rvpm add <github-path> [@version]` — append to `rv.toml`, resolve, write to `rv.lock`, populate cache. (v1 has a stub; gets the real implementation.)
- `rvpm install` — re-resolve `rv.toml` + `rv.lock`, populate cache. (v1 stub becomes real.)
- `rvpm update [path]` — bump pinned version in `rv.lock`.
- `rvpm build` — invokes the compiler with import resolution wired through the cache.
- `rvpm run` — `build` + execute.
- `rvpm fmt` — unchanged from v1 in spirit; the v2 formatter handles the new grammar.

### Security
- TLS via Git's HTTPS client (no custom transport).
- `rv.lock`'s tree-hash check guards against tag mutation.
- No execution of dependency code at install time (no `build.rs` equivalent in v2.0; can revisit).

## Verification

End-to-end verification per phase:

- **Phase 0 (branch):** `git checkout v2.x && git log` shows the branch; `gh pr` against `v2.x` triggers CI.
- **Phases 1-6 (frontend):** Per-stage golden snapshots committed in `tests/v2_*_golden.rs`. Each phase ships passing with its predecessors green.

- **Phase 9 (stdlib):** Every v1 example has a v2 counterpart whose stdout matches a hand-verified baseline under `examples/v2/*.rv.out`.
